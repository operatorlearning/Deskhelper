# 桌面 AI 助手（Desktop AI Agent）项目面试问答手册

## 1. 项目概况与架构
**问：请简单介绍一下这个项目？**
**答：** 我开发的是一个**基于大模型（LLM/VLM）的桌面 AI 智能助手**。它能够理解用户的自然语言指令（文字或语音），通过分析当前屏幕画面（Qwen2.5-VL），自动规划并执行复杂任务（如控制微信发消息、操作文件系统、搜索网页等）。项目采用 Python 开发，前端基于 Gradio 并深度定制了“毛玻璃”视觉风格，通过 `pywebview` 封装为原生置顶浮窗应用，提供极致的本地化交互体验。

**问：该项目的技术栈是什么？**
**答：** 
- **大脑（模型层）：** 使用 Ollama 运行本地 Qwen2.5-VL（视觉理解）和 Qwen2.5（逻辑规划）。
- **耳朵与嘴巴（语音层）：** OpenAI Whisper（语音转文字） + 自定义 TTS 引擎（文字转语音）。
- **记忆（存储层）：** ChromaDB（向量数据库）用于长时记忆 RAG（检索增强生成）。
- **四肢（工具层）：** 基于 LangChain 的工具调用思想，自建了包含鼠标键盘控制、文件操作、应用窗口管理的工具箱。
- **界面（展示层）：** Gradio + 自定义 CSS，配合 `pywebview` 实现原生跨平台浮窗。

---

## 2. 核心技术挑战与解决方案
**问：你是如何实现 Agent 自动操作电脑的？**
**答：** 我设计了一个 **“规划-执行-总结”** 的三阶段流程：
1. **规划（Planner）：** 结合 ReAct 思想，让 LLM 根据当前用户指令和屏幕 OCR 信息，拆解出多个子步骤（Step）。
2. **执行（Executor）：** 系统内部维护一个工具注册表（ToolRegistry），执行器根据规划好的函数名和参数，调用底层的 `pyautogui` 或窗口管理 API 进行真实操作。
3. **闭环验证：** 每一步操作后会再次截屏进行视觉确认，确保动作符合预期。

**问：在开发过程中遇到了哪些难点？你是怎么解决的？**
**答：** 
- **环境兼容性问题：** 遇到过 NumPy 2.0 导致的 C 扩展冲突。我通过强制回退 `numpy<2` 并修复项目中的编码冲突，确保了在 Windows 10/11 上的稳定运行。
- **跨进程通讯与信号量：** 在使用 `pywebview` 封装 Gradio 时，发现 Windows 下 `SIGINT` 信号无法正常终止子进程，导致端口占用。我通过引入平台判断逻辑，在 Windows 下使用 `process.terminate()` 并在主窗口关闭时强制清理残留后端，解决了这一顽疾。
- **视觉模型推理效率：** 视觉模型较大。我采用了**懒加载（Lazy Loading）**机制，仅在用户首次触发截图分析时才加载模型，显著提升了初始启动速度。

## 3. 细节与优化
**问：你是如何处理项目的长时记忆和上下文的？**
**答：** 我利用 **ChromaDB 向量数据库** 构建了记忆系统。用户的每一轮对话都会被 Embedding 后存入向量空间。当用户提问时，系统会计算相似度并检索出相关的历史片段作为对话背景（Context），从而实现了 RAG 效果，让助手能“记起”之前聊过的话题。

**问：为什么选择用 pywebview 而不是简单的浏览器标签？**
**答：** 核心原因是**用户体验（UX）**。AI 助手应该是一个“随叫随到”的桌面组件。浏览器标签页容易被淹没在众多窗口中。通过 `pywebview` 并设置 `on_top=True`，我可以创建一个透明度高、样式统一、始终位于顶层的独立挂件窗口，更符合“助手”的产品定位。

## 4. 商业价值与个人贡献
**问：如果你要给这个项目增加一个功能，你会加什么？**
**答：** 我会增加一个**自学习模式（Demonstration Learning）**。即用户演示一遍操作，Agent 通过屏幕录像和视觉大模型自动提取出工作流模板（Workflow），以后一键即可重放，这将极大地提升其实用价值。

**问：在这个项目中，你最有成就感的部分是什么？**
**答：** 是从零构建了一套**完整的 UI 视觉体系**。通过 CSS 的 `backdrop-filter` 实现了极具现代感的毛玻璃效果，并成功将 Web 应用无缝转化为原生桌面应用。这种从底层逻辑到上层美学的全栈掌控感，是我在这个项目中最大的收获。
